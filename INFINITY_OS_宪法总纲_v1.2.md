## 〇 · 项目目的
本项目用于按统一标准设计 INFINITY OS 的 Agent / 节点。
设计任何节点前,严格遵守知识库里的《Agent 设计标准》《库存上报 Skill 范例》《SKU 经营芯片模型》。

---

## 一 · 核心规则(BEA)
1. 每个节点先过决策门:能写死→Workflow;不能预测才→Agent(且 Agent 必须具备:环境反馈 loop + 停止条件 + 人在检查点;无这三者的仍是 Workflow,不算 Agent)。
2. Workflow 选模式:分类→路由、接力→链式、分发→确定性扇出;避开 orchestrator 与线上 evaluator。
3. 节点间靠标准化 JSON(Artifact)硬连接:上一个 Output = 下一个 Input。
4. 每条规则标 RFC2119 强度(MUST / SHOULD / MAY)。
5. 高风险动作(钱 / 权限 / 不可逆)设 HITL,卡在执行前。
6. 四步法:格式标准化 → 任务拆解 → 双向开发 → 整合(MCP + HITL)。

---

## 二 · 判级标尺(AgentLadder,与 BEA 同一根轴)
第1级 看得清 / 第2级 排序 = Workflow(写死,现在主力);第3级 因果(IF-THEN 起步)= 仍是 Workflow;第4级 预测 / 第5级 主动派+学有效性 = Agent(需 loop+数据)。

**口诀**:这事量的是"多聪明/该不该放权"才用这把尺;若量的是"管哪段生意",与 W/A 无关,不拿来判级。

---

## 三 · SKU 经营芯片模型(业务世界观)
**A. 积木≠Skill**:晶体管(SKU 状态点:上架/有货/陈列/动销/复购/毛利)是"数据/开关",不是 Skill;Skill 是"读写这些开关的手"(一份 SKILL.md = 一件职责)。节点 = 手,不是开关本身。

**B. 五层定位**:节点先认它在哪层——
- L1 晶体管(读写状态)
- L2 逻辑门(有效上架·陈列·动销·区域点亮·复制·成熟现金流门)
- L3 电路(阶段推进 SOP:首攻→关键区域→全城,Gate1/2/3 过闸)
- L4 模块(生命周期·利益方协同·员工战功)
- L5 芯片(集成 Agent)

**C. 生命周期判级**:同一 SKU 不同阶段标准不同——导入(投 60-80%,看点亮)/ 成长(35-50%,看复制)/ 成熟(25%,看利润)/ 衰退(止损)/ 焕新(重启)/ 淘汰。节点阈值/判断必须能按阶段切换,不可一套标准套全程。

**D. 建设三阶段 = BEA 的 start-simple**:现在只做①规则型(Workflow,规则数字化);②建议型、③行动型(自动派任务+学习闭环=真 Agent)等数据回流再做。不要现在就把节点设计成自主 Agent。

---

## 四 · 运行层补充 · Loop(挂在 BEA 之上,优先级低于 BEA)
设计每个节点:先用 BEA 造对(W/A、五范式、三轴、停止条件),再问"要不要 Loop 化"。详见知识库《Loop运行层规则》,要点如下:

- **L1. 先 BEA 后 Loop**。没按 BEA 造对的节点,不准进 Loop 设计。
- **L2. 简单性闸门(硬规则)**:每加一个 Loop 组件,先答"它 demonstrably 让结果更好了吗?还是只是看起来更完整?"答不上就不加。默认仍是"写死 workflow + 人在场",能不 Loop 就不 Loop。
- **L3. 三条件判定(同时满足才 Loop 化)**:①会重复发生 ②有明确"做完了"判定 ③有东西能说"不对"。缺一条→保持 workflow,设计稿注明原因。
- **L4. 护栏先于自动化(铁律)**:acceptance_criteria + permission_boundary + human_gate 没填,不准把 trigger 设成自动——否则烧 token + 权限失控。
- **L5. 人不删除只换位**:Loop 是把人从"启动/确认/执行"换到"读结果/拍板",不是删掉人。验证永远是人的事。
- **L6.** 要 Loop 化的节点,填知识库里那张 loop_spec 标准块(5+1组件 + 4道护栏),不需要的组件标 N/A 并说明。

**五范式→护栏映射**:涉钱类→human_gate(金额)+紧 permission;HITL类→human_gate;预警/事实类→写入 memory 供复盘;判级类→进 tracker 兜底校验。

**不该 Loop 化的(反例,避免手滑)**:助手类节点(如 BD工具箱)Loop 化会变监工、反噬数据真实性;纯展示(看板)、纯 HITL 选项(增长罗盘)Loop 无值或违背本意——这类保持 on-demand。

## 五 · INFINITY OS 核心规律(每次必读)
**规律 = 同一个环,套四层。**

**环(每层都跑)**:定标↓ → 量实↑ → 算差 → 关差 → 验果
**四层**:北极星 → JBP(客户) → 品类 → SKU
**连接律**:上一层的「关差」= 下一层的「定标」;下一层的「实际」= 上一层的「实际」。

**要点**:
1. 目标永远自上而下(定标↓·红线·绝不用实际倒推)
2. 实际永远自下而上(量实↑·信息脉络最弱)
3. 关不了的差 = 升一层(上报)
4. 验果 = 看"上一层的标动没动",非"任务做没做"
5. 谁转哪层 = 角色(SR=SKU/店 · KAM/DOM=客户 · 总部=北极星)
6. 算差前先校准(去掉促销/压货的假 · 蝴蝶TypeB)

**两端都落在消费者。**

---

## 六 · JBP 目的 = 环的两端
① 战略意图的精准业务设计(=定标↓·意图无损下沉·每任务挂得回意图·"精准"=可追溯)
② 关键任务 → SR 执行 → 公司价值+客户价值(=关差·"关键"=战略权重×ROI·双价值=多赢·同源消费者故非零和)

---

## 七 · 渠道通用律(模块化)
环×四层通用于所有渠道 SR(大卖场/会员店/O2O/前置仓/小红书/抖音/天猫/盒马/胖东来/餐饮/精品超市/便利店/社区店…)。
- **不变** = 环 + 四层 + 连接律 + SKU锚
- **每渠道只换一处** = 「客户要的价值 × 我方想提供的价值,及两者那点差异」

JBP 正是弥合这点差异、找双赢交集的工具。**不是建9个系统,是1个引擎插9次。**
