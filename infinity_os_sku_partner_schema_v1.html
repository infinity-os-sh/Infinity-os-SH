SECTION 01

核心定理 · 为什么 SKU 必须 合伙人化

这是整个 INFINITY OS 上层 Agent 体系成立的前提。没有这一层，所有 Agent 都是无主之物。

问题: 当前 INFINITY OS 把所有 SKU 当统一对象处理

心跳公式 a 值算法不区分 SKU 角色；蝴蝶效应所有 SKU 同等检测；Omakase 推单按整体心跳推；Uber 激励 CB 分对所有 SKU 同权重；经销商考核所有 SKU 视为相同。

结果: 六月鲜 380ml (北极星 Flagship) 和小康酱油 (防御品 Defensive) 在系统里被同等对待——这违反合伙人制 V1.2 的"差异化资源分配"原则。

⚠ 这是 INFINITY OS 的灵魂级缺失
合伙人制 V1.2 写了"客户合伙人 (双轨制)"，但其镜像在产品端的实现 = SKU 合伙人化。
没有 SKU 合伙人化，"4P 中 Product = 财务锚点"只是口号。
没有这一层，智能补货 / 蝴蝶 / 心跳 等所有上层 Agent 都缺少"主人"——无人负责、无人受益、无人学习。

解: 每个 SKU = 一个合伙人节点

每个 SKU 必须明确回答 6 个问题 → 6 张卡:

  1. 角色 Role · 你是哪种合伙人？
  2. 生命周期阶段 Stage · 你处于哪个阶段？
  3. 阶段目标 Goal · 你这个阶段要达成什么？
  4. 阶段投入 Input · 公司给你什么资源？
  5. 阶段产出 Output · 你必须回报什么？
  6. 退出条件 Exit · 什么时候降级或退场？
SECTION 02

7 种 SKU 角色 · 合伙等级体系

对应合伙人制 V1.2 的标准合伙 + 深度合伙双轨。资源向旗舰/主力倾斜——20% 深度合伙拿 80% 资源。

25%
DEEP S-LEVEL
🏆 旗舰 Flagship
北极星 SKU。深度合伙 S 级。极高资源倾斜。零容忍缺货。例:六月鲜核心规格。
30%
DEEP A-LEVEL
⭐ 主力 Core
战略品。深度合伙 A 级。高资源倾斜。长期培养。例:六月鲜其他主销规格。
25%
STANDARD B-LEVEL
📦 跑量 Volume
标准合伙。规则化运营。中等资源。按销量分配。例:味达美主销品。
8%
STANDARD C-LEVEL
🎯 利基 Niche
特定场景。中-低资源。定向投入。例:禾然有机 / 葱伴侣。
5%
TACTICAL D-LEVEL
🛡 防御 Defensive
阻击竞品。按需启动战术合伙。低投入按需调用。例:小康低价线。
5%
OBSERVATION
🌱 候选 Candidate
观察期。新品 6 月内测试预算。需审批。表现好转 Core/Niche，差则 Sunset。
2%
EXIT TRACK
🌅 退场 Sunset
终止合伙。减投资观察 6 月清库存。不再补货。替代品衔接。
资源分配原则 (合伙人制 V1.2 第六条对应)
20% 深度合伙 SKU (Flagship + Core) 拿 80% 资源——这是合伙人制的差异化分配原则在 SKU 层的实现。 不强制升级，不放弃标准合伙，但资源用结果分配。INFINITY OS 本身就是 SKU 合伙人筛选器——市场用销量+CB分投票。
SECTION 03

7 个 生命周期阶段

每个 SKU 在不同阶段有不同的目标-投入-产出组合。阶段切换由数据 + 业务团队 + 战略层三方共识决定。

阶段 特征 阶段目标 KPI 投入侧重
🥚 孵化 Incubation 上市前 / 测试中 研发达成 / 内测合格率 / 包装确认 R&D / 包材 / 试销点准备
🐣 引入 Introduction 上市 0-6 月 铺货率 / 试用率 / 复购率 激进铺货 / 高频促销 / KOC 种草
📈 成长 Growth 6-24 月 · 销量爬升 市占率达 X% / 城市覆盖 / 二批商数 渠道扩张 / 品牌投放 / 产能爬升
🌟 成熟 Maturity 稳定期 · 主力贡献 现金利润 / 毛利稳定 / CB 分 稳定补货 / 维护陈列 / 防御市占
🛡 防御 Defensive (阶段) 受到竞争压力 (任何角色都可能进入) 市占不流失 / 价格守住 / 客户保持 反击促销 / 价格补贴 / KA 锁定
📉 衰退 Decline 销量持续下滑 体面退场 / 库存清理 / 替代衔接 减少投放 / 清库促销 / 新品引导
🌅 退场 Exit 主动下线 库存归零 / 渠道撤回 / 法务结清 停产 / 清库 / 渠道沟通
⚠ 命名警告 · "Defensive" 同时是角色名和阶段名
角色 defensive = 战术合伙类型,SKU 的长期身份(例:小康低价线天生就是阻击品)
阶段 defensive = 当前状态,SKU 的临时处境(例:六月鲜旗舰品受竞品冲击进入防御阶段)

代码实现时必须用 namespace 区分:SkuRole.DEFENSIVE vs SkuStage.DEFENSIVE, 数据库字段也分别定义在 role_type 和 current_stage 列。 V1.1 计划将阶段重命名为 defending 或 embattled 彻底消除歧义。

关联表: SKU ↔ 合伙人 (多对多)

每个 SKU 关联多个合伙人。每个合伙人也关联多个 SKU。这是个人收入/绩效/奖金/晋升绑定的核心数据结构。

interface SkuPartnerLink { sku_id: string; // 关联 SKU partner_id: string; // 关联人 (员工/经销商/美顾) role_in_sku: string; // PM/品牌/KA/区域/美顾/经销商 income_share_pct: number; // 收入挂钩占比 (0-100%) kpi_weight: number; // 绩效考核权重 bonus_pool_share: number; // 奖金池分配比例 promotion_factor: number; // 晋升评估因子 started_at: Date; // 关联开始时间 ended_at: Date | null; // 关联结束时间 (退出后) }

SECTION 07 · 重点

智能补货的 5 个对接点 · 跨对话契约

这是 Path C 的核心。智能补货代码改造时必须实现/调用这 5 个接口，否则就是"无主孤魂"。

TYPESCRIPT INTERFACE
/** * INFINITY OS SKU 合伙人化框架 V1.0 - 智能补货对接接口 * 文档地址: https://infinity-os-sh.github.io/Infinity-os-SH/sku_partner_schema_v1.html * 任何上层 Agent (智能补货 / 蝴蝶 / 心跳 / 派单互锁) 都引用这份契约 */ interface SkuPartnerSchema { // 对接点 1: 获取 SKU 角色 (决定补货激进度基线) getRole(skuId: string): SkuRole; // 返回: 'flagship' | 'core' | 'volume' | 'niche' | 'defensive' | 'candidate' | 'sunset' // 对接点 2: 获取生命周期阶段 (决定补货节奏修饰) getStage(skuId: string): LifecycleStage; // 返回: 'incubation' | 'introduction' | 'growth' | 'maturity' | 'defensive' | 'decline' | 'exit' // 对接点 3: 获取补货策略 (核心配置 - 由 role + stage 派生) getReorderPolicy(skuId: string): ReorderPolicy; // 对接点 4: 获取关联合伙人 (推单给谁、奖金算给谁) getOwners(skuId: string): Partner[]; // 返回: [{partner_id, role_in_sku, income_share_pct, contact_method}] // 对接点 5: 获取阶段 KPI (决定补货量上限/下限) getKpiTargets(skuId: string): KpiTargets; } interface ReorderPolicy { reorder_threshold: number; // 触发补货的水位倍数 a aggressiveness: 'high'|'medium'|'low'; stockout_tolerance: 'zero'|'low'|'medium'|'high'|'full'; max_overstock: number; // 最大库存倍数 a approval_required: boolean; stage_modifier: number; // 阶段修饰因子 (Growth=1.2, Decline=0.8...) }
智能补货改造前后对比
改造前 · 算法仅依赖库存数据:
计算补货量 = f(库存水位, 销售率 r, 在途库存)

改造后 · 引入 SKU 合伙人化:
计算补货量 = f(库存水位, r 值, 在途, SkuRole, Stage, ReorderPolicy)
推送给 = SkuPartnerSchema.getOwners(skuId)
奖金回流 = 执行结果 × income_share_pct → 自动算入合伙人月度奖金

SECTION 09

生命周期 → 补货策略修饰

阶段不直接改变补货策略,但作为修饰因子叠加在角色基线上。

阶段修饰因子具体逻辑
🥚 Incubation预测驱动按销量预测 forecast 备货,不依赖 r 值 (无历史)
🐣 Introductionaggressiveness +1 级抢铺货率,即使 r 值不稳也激进补货
📈 Growthreorder_threshold × 1.2跟随销量增长率动态调高水位
🌟 Maturity默认 (1.0)按角色基线执行,无修饰
🛡 Defensivemax_overstock + 0.5a多备库存防止竞品填补缺口
📉 Declineaggressiveness -1 级降低补货激进度,清旧库
🌅 Exitreorder_threshold = 0停止补货,清场
最终补货量公式 V1.0
final_reorder_qty = base_a × role_threshold × stage_modifier - current_stock - in_transit

其中:
• role_threshold 来自 Section 08 表
• stage_modifier 来自 Section 09 表
• base_a 来自心跳系统 a 值校准 Agent

SECTION 12

风险与 约束

按"真实"原则,必须坦诚说出。

① 公平性问题

不同 SKU 难度不同。Flagship 容易出业绩,合伙人收入高;Defensive 天生收入低。
缓解方案: 角色加权算法 + 战略价值奖金 (战略价值 ≠ 销售价值)。

② 责任分离问题

SKU 失败究竟是产品问题、渠道问题、还是促销问题?谁担责?
缓解方案: 6 张卡的 KPI 必须按合伙人角色拆分(PM 负责产品 KPI, KA 负责渠道 KPI),不要混淆。

③ 跨周期问题

SKU 生命周期 5-10 年,但合伙人收入要月度结算。引入期投入大产出小,该怎么算?
缓解方案: 引入"延迟兑现"机制——引入期投入按战略奖金计入,成熟期产出按销售奖金兑现。

④ 退出机制问题

合伙人不达标怎么退?SKU 退场怎么处理关联合伙人?
缓解方案: 6 张卡的 Card 06 (Exit) 提前定义退出条件 + 退出协议。法务介入。

⑤ SKU 数量爆炸问题

五品牌 × 多规格 = 几百个 SKU。全部填 6 张卡工程量巨大。
缓解方案: 二八法则。先做 Top 50-80 战略 SKU 的完整角色卡。其余按"默认规则"(销量自动归类 Volume / Niche / Sunset)处理。

⚠ Claude 的核心警告
SKU 合伙人化是组织变革,不只是技术建设。需要 HR / 财务 / 法务 / 文化四件事并行,缺一不可:
• HR: 从职级制 → 合伙人制
• 财务: 从职能成本中心 → SKU P&L
• 法务: 合伙人协议体系
• 文化: 从职员心态 → 合伙人心态

INFINITY OS 解决不了 70% 的组织问题,需要业务层面共识。这些问题不要指望系统建设就能搞定。
SECTION 13 · 关键

状态机 · 角色和阶段的转换逻辑

SKU 不是静态的。市场在变,竞品在变,渠道在变。SKU 角色卡必须能随之转换,否则就是死卡片——这是机制层不是概念层的关键差异。

13.1 角色转换矩阵 (Role Transition)

合法转换路径:

Candidate → Volume / Niche / Sunset // 出生分流
Volume → Core / Niche / Sunset
Core ⇄ Flagship // 升降级
Niche ⇄ Volume
任何角色 → Defensive → 原角色 / Sunset // 受冲击临时切换
任何成熟角色 → Sunset // 退场

非法转换(分两类):

硬约束 (永远不允许,任何级别都不能批):

  • Sunset → 任何角色 (退场不可逆,需重新走 Candidate 重生)
  • 任何 → Candidate (Candidate 是出生角色,不可逆向)

软约束 (默认禁止,但 Tier 3 战略层可批准跨级):

  • Flagship → Sunset (默认禁止;Tier 3 可批且必须经 Defensive 缓冲期)
  • Niche → Flagship (默认必须 Niche → Volume → Core → Flagship 阶梯;Tier 3 可批"加速通道",仍走阶梯但每段时间窗口缩短)
  • Candidate → Flagship (默认禁止新品跳到旗舰;Tier 3 可批战略性新品破格)

13.2 阶段转换矩阵 (Stage Transition)

🥚 Incubation → 🐣 Introduction // 上市发生
🐣 Introduction → 📈 Growth // 铺货+复购达标
📈 Growth → 🌟 Maturity // 增速放缓+市占稳定
🌟 Maturity ⇄ 🛡 Defensive // 竞品冲击/威胁解除
🌟 Maturity → 📉 Decline // 销量持续下滑
📉 Decline → 🌅 Exit // 战略决策

13.3 转换触发规则 (具体阈值 + 时间窗口)

转换触发条件时间窗口决策权
Candidate → Volume 月销 ≥ 10万 + 复购率 ≥ 25% 持续 3 月 Tier 2 业务
Candidate → Sunset 月销 < 5万 或 复购率 < 15% 上市 6 月内 Tier 1 自动
Volume → Core 市占率 ≥ 5% + 战略评分 ≥ 60 持续 12 月 Tier 2 业务
Core → Flagship 战略层提名 + 北极星贡献度 ≥ 70 战略评估周期 Tier 3 战略
Maturity → Defensive 竞品价格降 ≥ 10% 或 市占率单月降 ≥ 2pt 立即触发 Tier 1 自动
Defensive → Maturity 威胁解除 (竞品价格回升 / 市占率恢复) 持续 3 月 Tier 2 业务
Maturity → Decline 销量同比降 ≥ 15% 持续 6 月 Tier 2 业务
Decline → Exit 销量同比降 ≥ 30% 持续 12 月 Tier 1 自动

13.4 决策权三级 (Decision Authority)

TIER 1
自动转换

规则触发,系统自动执行。
例:Candidate 6月未达标 → Sunset; 竞品冲击 → Defensive。

TIER 2
业务团队审批

规则推荐,业务批准。
例:Volume → Core; Defensive → Maturity 恢复。

TIER 3
战略层审批

战略决策。
例:任何 → Flagship; 主动 Sunset; 跨级跳跃。

13.5 历史快照表 (inf_sku_partner_history)

每次转换写入快照,完整可追溯。

13.6 转换冲突处理

  • Role 和 Stage 同时变化: 先处理 Stage 转换,再处理 Role 转换 (阶段是更稳定的属性)
  • 跳级转换: 必须 Tier 3 战略层审批,且必须填写完整理由记录到 trigger_data JSON
  • 角色阶段冲突 (例:Flagship 进入 Decline): 触发 Defensive 紧急复盘 + 6 月观察期,不立即降级
  • 循环转换防护: 同一 SKU 每月最多 2 次转换 (避免抖动)
🔗 智能补货必须订阅状态变化事件
智能补货代码每次执行前必须重新调用 getRole() 和 getStage(),缓存不超过 1 小时。 或者订阅 sku_state_changed 事件,在转换发生时立即清空旧策略缓存。

反例: 智能补货启动时一次性加载所有 SKU 的 role/stage 缓存到内存,然后跑一整天—— 这会导致状态机变化后补货决策仍按旧策略执行,违背设计初衷。
SECTION 14 · 关键

奖金计算公式 · 合伙人制实质

如果说 Section 04-09 是 SKU 合伙人化的"数据结构",这一节就是"利益结构"。没有这一节,合伙人制 V1.2 就只是口号。

14.1 月度奖金的核心结构

奖金分两部分:个人份额(按 income_share_pct 计算,≤70%) + 团队池分配(30% × 团队系数)。这两部分是叠加关系,不是相乘。

月度奖金 = 基本工资
+ 个人 SKU 奖金 // = Σ(单 SKU 个人销售奖金 + 单 SKU 个人战略奖金)
+ 团队池分配 // = SKU 团队池 × 团队系数(0.7-1.3) × 个人在团队中的占比
+ 上限/下限调整 // 见 14.5
边界: 总奖金部分 ∈ [基本×20%, 基本×200%], 超出进入季度池

💡 这里的"个人 SKU 奖金"和"团队池分配"是独立两条管道:个人按 income_share_pct (70% 上限) 拿走个人份额,剩下的 30% 进 SKU 团队池,池子按团队 KPI 系数 (0.7-1.3) 调节后再二次分配给团队成员。这样设计避免了"个人独吞"也避免了"大锅饭"。

14.2 个人 SKU 销售奖金公式

单 SKU 个人销售奖金 = max(0, 实际现金利润 − 阈值)
× 角色系数 // Flagship 1.5x / Core 1.2x / Volume 1.0x / Niche 0.8x
× 阶段修饰 // Growth 1.2x / Maturity 1.0x / Decline 0.8x / Defending 1.1x
× income_share_pct // 个人份额 (0-100%, 注:全员合计 ≤ 70%)
× KPI 达成系数 // 见下表(线性插值)

KPI 达成系数(线性插值):

KPI 达成率系数说明
< 80%0不发奖金 (低于及格线)
80% - 100%0 → 1.0线性插值: 系数 = (达成率 − 0.8) / 0.2
100% - 120%1.0 → 1.2线性插值: 系数 = 1.0 + (达成率 − 1.0)
> 120%1.2 (上限)封顶,避免短期投机性冲量

角色销售系数:

角色销售系数说明
🏆 Flagship1.5x北极星 SKU,激励最高
⭐ Core1.2x主力品,高激励
📦 Volume1.0x跑量品,标准激励
🎯 Niche0.8x利基品,激励较低
🛡 Defensive1.0x防御品,等同 Volume
🌱 Candidate0不发销售奖金,只发战略奖金
🌅 Sunset0不发奖金,仅基本工资

14.3 SKU 战略奖金公式 (解决跨期问题)

引入期投入大产出小,Candidate / Introduction 阶段如果只算销售奖金,合伙人收入会为负。战略奖金机制保护这些阶段的合伙人。

单 SKU 个人战略奖金 = 战略价值评分 × 战略奖金基数 × income_share_pct

💡 分配规则一致性:战略奖金与销售奖金同一个 income_share_pct,即合伙人所有奖金同比例分配。 剩余 30% 进 SKU 团队池(与销售奖金团队池合并),由团队系数二次分配。这保证奖金机制简洁、可预测。

战略价值评分 (0-100,季度评估):

维度权重评估标准
北极星贡献度40%是否带动六月鲜 200M 瓶 / 4B RMB 北极星
品牌心智占领25%CB 分提升 / 品类地位
渠道战略价值20%是否打开新渠道 / KA 战略客户
创新引领15%是否开创品类细分 / 技术突破

战略奖金基数 (按角色):

角色战略奖金基数 (元/月)说明
🏆 Flagship¥50,000北极星 SKU 战略价值最高
⭐ Core¥30,000主力品
📦 Volume¥10,000跑量品
🎯 Niche / 🛡 Defensive¥5,000利基/防御
🌱 Candidate¥10,000跨期保护:即使无销售也发战略奖金
🌅 Sunset¥0退场期不发
💡 跨期保护机制 (核心创新)
Candidate / Introduction 阶段强制发 100% 战略奖金,即使该 SKU 当月销售为 0。
理由: 引入期是为未来投入,合伙人不该因此收入为负。这是合伙人制 V1.2 "Alignment" 的关键体现——把短期个人利益和长期组织战略对齐。

14.4 团队系数与个人/团队分配

  • SKU 个人合伙人份额 (income_share_pct 总和) ≤ 70%
  • 剩余 30% 进 SKU 团队池
  • 团队池按团队整体 KPI 达成率 (DOM 评定) 二次分配
  • 团队系数 = 团队 KPI 达成率 (0.7 - 1.3 区间)

14.5 上限 / 下限 / 保底

项目规则目的
月度保底奖金 ≥ 基本工资 × 20%避免短期波动伤合伙人
月度上限奖金 ≤ 基本工资 × 200%避免单月暴富,平滑分配
季度调节季度内可平滑跨月解决月度波动
年度清零年终结算清零避免无限累积

14.6 离职 / 调岗 / 新增协议

事件处理规则
离职 已确认的销售奖金按比例兑现 6 个月; 战略奖金一次性结清; 未来奖金归零; link.ended_at 设为离职日
调岗 link.ended_at = 调岗日; 新岗位重新签 link; 旧岗位本月奖金按时间比例计算 (天数 / 当月天数)
新增 从加入月起算; 前 3 个月按 50% 折算 (适应期); 第 4 月起 100%
晋升 晋升到新角色的 income_share_pct 比例可能调整,需走 HR 流程

14.7 实例:张三 5 月奖金计算

▼ 案例 · 六月鲜 380ml PM 张三 (基本工资 ¥20,000)

关联 SKU:六月鲜 380ml (Flagship/Maturity), income_share_pct = 30%, KPI 达成 105%

① 个人 SKU 销售奖金:
= max(0, 实际利润 ¥1,000,000 − 阈值 ¥800,000)
× 1.5 (Flagship) × 1.0 (Maturity) × 30% (份额) × 1.05 (KPI 105%)
= ¥200,000 × 1.5 × 1.0 × 0.30 × 1.05 = ¥94,500
② 个人 SKU 战略奖金:
= (战略评分 75/100) × ¥50,000 (Flagship 基数) × 30% (份额)
= 0.75 × ¥50,000 × 0.30 = ¥11,250
③ 团队池分配 (独立管道):
SKU 团队池 = 该 SKU 总奖金 × 30%
= (¥315,000 + ¥37,500) × 30% / 团队人数 × 团队系数 1.05 (DOM 评定)
张三在团队中的占比 1/3 → 团队池分配 ≈ ¥37,000
注:¥315,000 = 销售奖金原始(未乘 income_share_pct)的 70% 个人份对应的 30% 团队池基数
奖金部分原始合计 = ① + ② + ③ = ¥142,750
④ 触发上限规则 (14.5):
奖金上限 = 基本工资 × 200% = ¥20,000 × 2.0 = ¥40,000
¥142,750 > ¥40,000 → 本月奖金封顶 ¥40,000
超出 ¥102,750 进入 Q2 季度池,6/7 月平滑分配
本月实发 = 基本 ¥20,000 + 奖金上限 ¥40,000 = ¥60,000
Q2 季度池累积 ¥102,750 (6/7 月平滑)
若张三 6/7 月业绩较差,季度池可补足保底 ¥4,000 (基本工资 20%)
季度结束未发完的余额年终清零

💡 这就是 14.5 上限/季度调节机制的实际作用:本月超额部分进入季度池,平滑跨月波动,避免单月暴富 (公平性) 也避免单月暴跌 (合伙人积极性)。年终季度池余额清零,确保激励周期不超过 1 年。

⚠ Claude 必须警告:奖金公式不只是数学,更是组织设计
上述公式是 V1.0 起始版本。实际落地需要 HR + 法务 + 财务三方共识:
• HR: 与现有薪酬体系兼容 (基本工资 + 奖金 vs 全包年薪)
• 法务: 合伙人协议法律效力 (劳动法 vs 合伙法)
• 财务: 现金流压力评估 (Flagship 高奖金的财务承受能力)
