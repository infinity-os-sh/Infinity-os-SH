## 核心设计原则

### ① 三源同等地位,按 weight 合并(非覆盖)
同一个晶体管信号(如 S01 上架)可能同时来自销售员录入、POS、图像 AI。**按可信度 weight 合并**,不是简单覆盖。S06 动销以 POS 权威,S01 上架以销售员权威,S09 老板推荐只能来自销售员(无其他源)。

### ② 信号有保质期(TTL)
关键信号(S02 库存、S06 动销、S08 断货)TTL = 24 小时,过期变"未知"必须重采集。**宁可标未知,不可用过期数据骗 Agent**。

### ③ 严格遵守视觉识别 3 阶段(记忆 #29)
图像 AI 字段强制有 `stage` 标记:`stage_1_manual` / `stage_2_ai_human_verify` / `stage_3_ai_default`。**当前默认 stage_1**,绝不可一步到位直接接受 AI 识别。

### ④ 业务规则校验在合规校验之外
JSON Schema 只能验结构。业务规则(GPS 围栏、时间合理、晶体管逻辑一致性)由 `validator.py` 完成。**校验不过的数据拒收**,不进 L0 队列。

### ⑤ 多源冲突喂 L8 学习层
销售员说"有动销"但 POS 显示 0 —— 不是 bug,是**喂给 L8 自主学习的金矿**。系统标记 `conflict=true` 保留两版,L8 学习"什么情况下销售员说的更准"。

## 这份契约 → L1 晶体管表 → L2 逻辑门

完整数据流,以六月鲜轻盐 500ml 在上海 A 社区店为例:

```
1. 销售员张三进店检查
   → 用 01_sales_rep_input 录入(see example_01_sales_rep_input.json)
   
2. validator.py 校验
   → Schema OK + 业务规则 OK + 无冲突 → 接收

3. ETL 按 04_l0_to_l1_mapping 规则
   → 合并多源 → 生成 L1 transistor_signals 表一行
   → {S01:✓ S02:✓ S03:✗ S05:✓ S06:✗ ... S10:✓}

4. L2 导入期生命信号门触发
   → 规则: S01 AND S02 AND S06 = 生命信号成立
   → 结果: FAIL(S06 ✗)
   → next_step: 进入导入期调整动作

5. L7 行动建议引擎
   → 输出: [试吃, 老板培训, 第二陈列, 短期促销]

6. L8 资源调度
   → 触发: 申请试吃资源·申请陈列费用·派任务给 SR

→ 一个完整 GT 社区店【经营细胞】闭环
```
