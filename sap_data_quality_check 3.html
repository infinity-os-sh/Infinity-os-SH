检查 #1 · 表是否都存在
P0 必须
7张SAP表是否都在Hologres里?
▸ 为什么 sap_req_001要的7张表 · 任何一张缺失 · 整个项目要重新规划
SELECT table_name FROM information_schema.tables WHERE table_schema = 'sap_data_lake' AND table_name IN ('mara','kna1','mb51','mb52','vbrk','fbl5n','lips');
▸ 期望结果 返回7行 · 每张表都存在 · 如果少 → 立刻找IT VP补
检查 #2 · 数据是否最新
P0 必须
最近1天的数据是否同步过来?
▸ 为什么 如果数据停在1周前·INFINITY OS就是"看历史"不是"看现在"·失去价值
SELECT MAX(BUDAT) AS last_date, COUNT(*) AS rows FROM sap_data_lake.mb51 WHERE BUDAT >= CURRENT_DATE - INTERVAL '7 day';
▸ 期望结果 last_date是昨天 · rows > 100万 · 如果last_date超过3天前 → IT同步任务出问题
CHECK 3-4 / 10
准确性 · 数据是否对得上
检查 #3 · MB51与VBRK对账
P0 必须
发货流水(MB51) vs 发票(VBRK) 数量差异?
▸ 为什么 正常情况两者基本一致 · 差异>5%说明有"发货未开票"或"开票未发货"·业务有问题
SELECT (SELECT SUM(MENGE) FROM mb51 WHERE BUDAT=CURRENT_DATE-1 AND BWART='601') AS shipped, (SELECT SUM(QUANTITY) FROM vbrk WHERE FKDAT=CURRENT_DATE-1) AS invoiced;
▸ 期望结果 shipped和invoiced数值接近 · 差异<5% → 通过 · 5-15% → 警告 · >15% → 有大问题
检查 #4 · 物料编号一致性
P1 重要
MB51的物料编号·MARA里都能找到吗?
▸ 为什么 如果MB51里有MARA没有的物料编号·说明数据不一致·后面JOIN会丢数据
SELECT COUNT(DISTINCT m.MATNR) AS orphan_skus FROM mb51 m LEFT JOIN mara r ON m.MATNR = r.MATNR WHERE r.MATNR IS NULL AND m.BUDAT >= CURRENT_DATE - 30;
▸ 期望结果 orphan_skus = 0 → 完美 · 1-5 → 可接受 · >5 → 找IT补MARA数据
CHECK 5-6 / 10
合理性 · 数据是否符合常识
检查 #5 · 异常大数检测
P1 重要
单笔发货数量是否有异常巨大值?
▸ 为什么 单笔发1000万瓶 · 99%是录入错误(多输了一个0)·会污染Agent判断
SELECT MBLNR, MATNR, MENGE, BUDAT FROM mb51 WHERE BUDAT >= CURRENT_DATE - 30 AND MENGE > 100000 -- 单笔超过10万瓶 ORDER BY MENGE DESC LIMIT 10;
▸ 期望结果 返回0行 → 完美 · 返回少数行 → 找业务确认是否真实订单 · 大量行 → SAP数据有质量问题
检查 #6 · 负数检测
P1 重要
数量字段是否有负数?
▸ 为什么 MENGE应该>0 · 负数通常是退货(应该用601对应的退货类型) · 如果601带负数·业务规则不对
SELECT BWART, COUNT(*) AS cnt, SUM(MENGE) AS total FROM mb51 WHERE MENGE < 0 AND BUDAT >= CURRENT_DATE - 30 GROUP BY BWART;
▸ 期望结果 只有退货类型(602)有负数 · 如果601有负数 → 业务规则要梳理
CHECK 7-8 / 10
关键字段空值检测
检查 #7 · 客户编号空值
P0 必须
MB51里发货记录·客户编号是否齐全?
▸ 为什么 没有客户编号·INFINITY OS无法判断这笔货发给谁·心跳计算无意义
SELECT COUNT(*) AS total, COUNT(KUNNR) AS has_customer, COUNT(*) - COUNT(KUNNR) AS missing_customer FROM mb51 WHERE BUDAT >= CURRENT_DATE - 30 AND BWART='601';
▸ 期望结果 missing_customer = 0 · 缺失率<1% → 通过 · >1% → 找业务确认
检查 #8 · 经销商地区空值
P1 重要
KNA1里经销商的省/市字段·是否齐全?
▸ 为什么 INFINITY OS的"上海·三省·山姆"分析全靠地理字段·没有就废了
SELECT COUNT(*) AS total, SUM(CASE WHEN REGIO IS NULL THEN 1 ELSE 0 END) AS missing_region, SUM(CASE WHEN ORT01 IS NULL THEN 1 ELSE 0 END) AS missing_city FROM kna1;
▸ 期望结果 missing都是0 · 缺失率<2% → 通过
CHECK 9-10 / 10
业务对账 · DS亲自验证
检查 #9 · 与业务方报表核对
P0 必须
SAP算出来的"上海昨天发货量" vs 业务团队的"日报数据"
▸ 为什么 这是最终极验证 · 数据再完整·和业务方对不上就是没用 · DS必须亲自做这一步
-- 算出昨天上海六月鲜的发货量 SELECT SUM(m.MENGE) AS shanghai_yesterday FROM mb51 m JOIN kna1 k ON m.KUNNR = k.KUNNR WHERE m.BUDAT = CURRENT_DATE - 1 AND m.BWART = '601' AND k.REGIO = '上海' AND m.MATNR LIKE 'LYX%';
▸ 期望结果 DS拿这个数 · 找上海销售总监问"昨天我们上海发了多少六月鲜" · 两个数误差<3% → 数据可信
▸ DS亲自验证 Tech Lead Day 1下午发数据给DS · DS15分钟找上海销售总监对一下 · 对得上 → SAP数据可用 · 对不上 → 必须查清楚为什么
检查 #10 · 时区一致性
P1 重要
SAP的时间戳是UTC还是北京时间?
▸ 为什么 时区错了·"昨天"算成"前天"·所有日期统计错位
SELECT MAX(BUDAT) AS sap_max_date, NOW() AS server_now, TIMESTAMPDIFF(HOUR, MAX(BUDAT), NOW()) AS hour_diff FROM mb51;
▸ 期望结果 hour_diff在合理范围(0-24小时) · 如果是8小时整或16小时整 → 时区问题
