01 / GIT WORKFLOW · Git协作流程
5人并行不冲突的关键
铁律 · 分支策略
分支命名 + Pull Request
▸ 主分支 main = 生产代码 · 永远可上线 · 任何人不能直接push ▸ 开发分支 每个任务开新分支 · 命名: [role]/[task]-[date]
data/sap-mb51-sync-0421 backend/inf-nodes-api-0421 wechat/push-template-0422 devops/ecs-init-0421
▸ 合并流程 开发完成 → 提Pull Request → Tech Lead+1人Review → 合并main → 删除分支
▸ 该做每天早上从main拉最新·避免落后
▸ 别做分支跑1周不合并 · 等合并时冲突
铁律 · Commit规范
Commit Message统一格式
▸ 格式
[type]: brief description (50字符内) 详细说明(可选·中文) - 关联Issue #123
▸ Type类型
feat: 新功能 fix: 修bug refactor: 重构(不改功能) test: 加测试 docs: 文档 chore: 构建/工具
▸ 示例
feat: 添加SAP MB51同步定时任务 - 每小时跑1次·增量同步 - 异常会写入inf_alerts - 关联Issue #15
02 / DAILY REPORT · 日报机制
每天17:00 · 1分钟模板
铁律 · 日报模板
5人每天17:00 · 群里发1分钟日报
▸ 模板(复制粘贴改数字)
📅 [姓名] · [日期]日报 ✅ 今天完成 1. [完成的事1] 2. [完成的事2] 🚧 进行中 1. [在做的事·完成度XX%] ⚠️ 阻塞(如有) 1. [遇到的问题·需要谁配合] 🎯 明天计划 1. [明天要做的事]
▸ Tech Lead要做的 每天17:30 · 5人日报到齐后 · Tech Lead发1条"全队日报汇总"给DS · 1分钟读完
▸ 该做简洁·1分钟读完·不写废话
▸ 别做"今天搞了一些事·明天继续搞"(无信息量)
03 / COMMUNICATION · 沟通规范
什么事走什么渠道
铁律 · 渠道分级
3个渠道·各管各的
▸ 企微群(实时讨论) 用于:今天的小事·临时同步·快速问答
不用于:重要决策·持久记录 ▸ GitHub Issues(决策+长期跟踪) 用于:技术方案讨论·Bug跟踪·任务分配·重要决策
所有决策必须落到Issues · 否则等于没决定 ▸ GitHub Pull Request(代码审查) 用于:Code Review · 代码相关讨论
不要在PR里讨论无关话题
铁律 · 求助原则
30分钟原则
▸ 卡住超过30分钟 · 立刻求助 不要闷头扛2小时 · 浪费的是公司时间 · 不是你的面子

▸ 求助格式
🆘 [你的名字]需要帮助 问题:[1句话说清楚] 我试过:[已经尝试的方案] 卡在:[具体卡在哪一步] @[最可能帮你的人]
▸ Tech Lead响应SLA 工作时间内 · Tech Lead必须15分钟内回应
04 / CODE STANDARDS · 代码规范
Java代码·5人统一
铁律 · 命名+注释
3条最低标准
▸ 命名 • 类名:大驼峰 AValueCalibrationAgent
• 方法名:小驼峰 calibrateNode
• 常量:全大写 MAX_RETRY_COUNT
• 包名:小写 com.xinhe.infinityos.agent ▸ 注释 • 复杂业务逻辑必须有中文注释
• Public方法必须有JavaDoc
• TODO必须带姓名+日期 // TODO 张三 0421 处理边界情况 ▸ 异常处理 • 不允许吃异常 catch(Exception e){}
• 必须log.error或抛出 · 让Tech Lead决定

出问题找谁 · 不要瞒

坏消息 · 早说 = 解决 坏消息 · 晚说 = 灾难 坏消息 · 不说 = 出局
