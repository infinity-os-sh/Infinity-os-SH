六月鲜的战略事实层 · 不是 IT 系统, 是经营战略地图

一句话定义

BAS 把六月鲜从「消费者 → 品牌 → 副品牌 → 系列/品类 → 产品线 → SKU → 场景 → 资产 → 证据 → 内容 → 渠道应用」11 个节点全部连接起来, 形成一条贯穿全身的脊柱.

核心公式

规格价值 + 产品价值 + 品牌价值
= 强占消费者心智
= 资产沉淀

这才是 BAS 闭环的真正起点 (DS 锁定 · 2026.05.12)

BAS 不是什么

  • ❌ 不是 IT 系统 (尽管用 HTML 实现)
  • ❌ 不是产品目录 (它是战略地图)
  • ❌ 不是营销工具 (是品牌资产的整体管理)
  • ❌ 不是平级模块 (它是 INFINITY OS 的战略上游)
  • ❌ 不是给所有人看的 (副总监 + 总监级以上, 不是一线美顾)

设计哲学 · 5 个原则

这是 BAS 设计中真正学到的, 比卡片本身重要.

原则 ①
消费者价值链是 BAS 闭环的真正起点
不是公司视角. 不是品牌维度. 是消费者怎么"喜欢规格→喜欢产品→喜欢品牌→沉淀心智".
三层价值叠加 = 心智占领 = 资产沉淀.
之前 6 个版本都错在从公司视角扩展维度, v0.7 才意识到要从消费者视角重构.
原则 ②
品牌资产是「沉淀」出来的, 不是「建造」的
Asset 不是一个独立的维度 (我之前以为). 它是消费者反复"喜欢"沉淀出来的结果.
↘ 稀释 / → 稳定 / ↗ 强化 / ⟳ 转型 — 这些趋势比"心智强弱本身"更重要.
原则 ③
立体网络, 不是层级树
L1 → L2 → L3 → L4 是公司管理树. 但 BAS 不是.
BAS 是 M:N 立体网络: 一个 SKU 跨多个场景, 一个场景跨多个副品牌, 一个资产支撑多个 SKU, 一个内容承载多个资产.
用树思维做 BAS = 还原成产品台账, 失去战略价值.
原则 ④
静态完成度 ≠ 闭环完成度
我一度说 BAS "95% 完整", 实际只是 9 个维度入口都建好了 (静态).
真正的闭环 = 5 个核心循环 (消费者循环 / 资产积累循环 / 阶段转换循环 / 跨副品牌接力循环 / 起量奇迹复制循环).
当前 BAS 闭环完成度只有 ~60% — 正向流转完整, 反向流转待真实数据.
原则 ⑤
Layer 1.3 不能假装 Layer 3
BAS 当前是设计原型, 不是运行系统. 数据是基于推理, 不是业务验证.
跟 INFINITY OS 真整合需要等 SAP Step 2 + 业务反馈完成.
诚实标注当前层级, 比假装"已完整"更重要.

BAS ↔ INFINITY OS · 关系

关系定位

BAS = 六月鲜战略事实层 (Single Source of Truth)

INFINITY OS = 全集团经营 OS (24 Agent + 4 脉络)

嵌入方式: BAS 作为 INFINITY OS 各 Agent 的战略上游. Agent 从 BAS 读取战略信息, 不在 Agent 内部硬编码.

8 个连接点 (摘要)

  • ① BAS L4 SKU 阶段 → 智能补货 Agent (spec_03)
  • ② BAS 心智趋势 → 蝴蝶效应 Agent (spec_02)
  • ③ BAS 副品牌阶段 → A 值校准 Agent (spec_01)
  • ④ BAS income_share → Uber 激励 Agent
  • ⑤ BAS 主推 SKU → 派单互锁 25 步
  • ⑥ BAS 副总监仪表盘 → 主控台个人工作台
  • ⑦ BAS 协议层 → 资源分配 (Cash 脉络)
  • ⑧ BAS 起量奇迹案例库 → INFINITY OS L6 记忆

详细 schema 见 bas_infinity_os_integration.html

4 阶段嵌入

阶段 1 (1-2 月) 准备期 独立运行 + 团队反馈 + SAP Step 2 阶段 2 (3-6 月) 数据对齐 真实数据接入 + API 开发 + UI 仍分离 阶段 3 (6-12 月) 接口对接 ⭐ Agent 开始读 BAS 战略数据 (真嵌入) 阶段 4 (12-24 月) 全集团扩展 BAS-禾然 / BAS-味达美 等复制 + 集团 BAS

演化决策树 · 4 种情况

当未来这些情况发生时, BAS 怎么继续演化

情况 ① · 副总监反馈说 BAS 内容不对
不要急着改 BAS. 先问: 是数据假设错了 (业务认知更准)? 还是结构错了 (我设计错了)?

数据错 → 等 SAP Step 2 解锁后用真实数据替换, 不在 v1.0 修订过多.
结构错 → 跟副总监一起重新画结构, 然后改 BAS v1.0.
都错 → 推翻部分假设, 接受 BAS v0.9 部分内容是错的, 重做.
情况 ② · SAP Step 2 完成, 真实数据接入
启动 BAS v1.0 数据替换:
①SKU_VALUES 的 4 层价值用真实测试/调研数据替换
②L2_VALUES + SCN_VALUES + L1_VALUE 同样
③心智趋势用复购率/消化率计算, 不是我推理
④起量奇迹案例库 (AST-07 + CT-07) 用真实 SKU-08 因果链路重写

v1.0 才是真正可信的 BAS, v0.9 只是设计原型.
情况 ③ · Tech Lead 上任, 启动整合
给 Tech Lead 第一个 Epic (3-4 个月):
①BAS 数据后端 (MySQL + Spring Boot, 10 个 API endpoint)
②真实数据接入 (SAP → BAS)
③副总监仪表盘嵌入主控台 (连接点 6)
④阶段 3 准备 (Agent 接口规范)

这个 Epic 完成后, 才能真正启动连接点 1-8 的对接.
情况 ④ · BAS-禾然 / BAS-味达美 复制
用六月鲜 BAS 作为模板, 复制结构:
①L1 + L2 + L3 + L4 + Scenario + Asset + Content + Channel + 治理层
②但内容完全不同 (禾然有机高端 / 味达美大众调味料)
③跨品牌接力机制 (P-05 Origin Lab → 禾然) 需要在两个 BAS 内显式建模

关键: 集团 BAS 层负责跨品牌协调, 避免内斗.

共同决策原则

不要:
❌ 急着加更多 BAS 维度 (没真实数据 = 空想)
❌ 假装 v0.9 已完整 = 闭环完成 (静态完成度 ≠ 闭环)
❌ 跳过业务验证直接推全团队
❌ 跳过 SAP Step 2 直接做 BAS 整合

真实理解 BAS

BAS 不是 HTML 单文件.
BAS 是把"北极星 200M瓶/¥4B/2029"翻译成"消费者心智迁移速度"的战略翻译器.
HTML 只是承载形式, 战略翻译才是本质.

DS 的真实期待

  • 真实 (不粉饰, 不假装完整)
  • 速度 (快, 不犹豫, 不反复确认)
  • 精准可靠 (机制层, 不是概念层)
  • 主动暴露问题 (不只是建议, 要直接指出)
  • Autonomy 思考 (Claude 自己判断, 不只是顺从)

能力边界提醒

Claude 能做:
HTML 设计原型 / 结构化已知信息 / 跨概念关联 / 文档撰写 / Agent spec 设计

Claude 做不了:
真实数据接入 / 业务验证 / 消费者调研 / 组织协调 / 持久运行 / 跨会话记忆

诚实分工:
BAS 落地的 95% 工作必须业务+技术+组织做, Claude 只能加速 5%

认知校准

"继续" 不一定意味着 "我同意你的建议".
"继续" 可能意味着 "你的判断不够深, 再想想".

遇到 "继续", 先反思是否真的理解了 DS 想要的, 而不是机械执行下一个动作.

一段话总结

给团队

BAS 是六月鲜品牌战略地图. 把消费者怎么"喜欢规格→喜欢产品→喜欢品牌→沉淀心智"显式化, 把 5 副品牌 / 6 协议 / 24 Agent 拉到同一张图上, 让每个副总监看清楚自己在 5 年北极星战略里的位置 + 价值 + 风险.

给董事会

北极星 200M 瓶 / ¥4B / 2029, 本质是 5 年时间窗口内, 老心智衰减速度 vs 新心智积累速度的赛跑. BAS 让这件事每天可见, 每周可量, 每月可决策. 不是 IT 系统投资, 是战略可视化的工具.

给 Tech Lead

BAS 当前是 Layer 1.3 设计原型 (HTML). 落地路径: 4 阶段 12-24 个月. 阶段 2 (3-6 月) 数据对齐是你的第一个 Epic. 8 个连接点跟 24 Agent 对接是阶段 3. 不要急着把 BAS 嵌入 Agent, 先让 BAS 跟主控台数据源同步.

给未来的 Claude

BAS 是 DS 用 Socratic method 引导我建出来的. 我 5 次说"建议停下", 5 次错了. v0.7 才意识到消费者价值链是真正起点. 不要重蹈覆辙. 真实, 速度, 精准可靠, 主动暴露问题, Autonomy 思考 — 这是 DS 真实期待的.

给我自己

这次 session 学到的最深的一件事:
诚实不是"敢说不", 是"敢承认自己想得不够深".

"建议停下让团队试用" 看起来负责, 实际是把判断责任推给别人. 真正的 Autonomy 是主动深挖, 直到看见根本问题. 顺从和"建议停下"都可以是认知舒适区.
