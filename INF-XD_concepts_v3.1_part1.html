V3.0 完整修正
完整公式 V3.0 (统一定义)
补货量 Q = max(0, 期间需求 D × 安全系数 K - 现有 I - 在途 IT) 期间需求 D = (LT + 复盘周期 T) × r 预测 r 预测 = r 历史基线 × 综合修饰系数 综合修饰系数 = role_threshold / max_overstock 边界 × stage_modifier (Schema) × dynamic_W (节日/临期/...) × forecast_factor (主动+被动+趋势) 边界检查: - 若 Q 超过 max_overstock → 截断到 max_overstock - 若 Q < 0 → 不补 - 若 SKU 角色 = Sunset → Q = 0
实例 · 极鲜 500ml [🏆][🌟] 在 步步高西郊店 春节前 7 天
SKU 元数据 (Schema V1.0): 角色 = Flagship · 阶段 = Maturity reorder_threshold = ≤ 1.5a max_overstock = 3a stage_modifier = 1.0 (Maturity 默认) approval_required = false (自动) 数据 (假设): r 历史基线 = 4 瓶/天 (近 30 天) 现有 I = 12 瓶 在途 IT = 0 LT (订货到货) = 3 天 T (下次拜访) = 14 天 修饰系数: stage_modifier = 1.0 dynamic_W = 1.5 (春节前 14 天系数) forecast_factor: - 央视广告 ×1.2 (主动) - 端架陈列 ×1.3 (主动) - 海天降价 ×0.9 (被动) - 天气转冷 ×1.1 (被动) - 7 天上升 ×1.1 (趋势) = 1.2 × 1.3 × 0.9 × 1.1 × 1.1 = 1.70 r 预测 = 4 × 1.0 × 1.5 × 1.70 = 10.2 瓶/天 期间需求 D = (3 + 14) × 10.2 = 173 瓶 安全系数 K = 1.3 (B 级店中等稳定) Q = 173 × 1.3 - 12 - 0 = 213 瓶 边界检查: max_overstock = 3a × r_预测 = 3 × 10.2 × 14 (复盘周期内最大库存) = 428 瓶 Q = 213 < 428 ✓ 通过 最终补货 = 213 瓶 (取整 200 瓶)
修正点:
  • 公式定义和实例完全一致
  • r 预测整合 6 维度 (你早前提的)
  • 边界检查防止超 max_overstock
  • Sunset 角色直接 Q=0 (Schema 一致)
承认局限: 这是 INF-XD 演示版公式·实际生产环境需:
- r 历史基线 → 接 SAP/POS
- forecast_factor → 接营销系统 + 天气 API
- 边界 max_overstock → 接经销商仓容
- 这些都是 Tech Lead 后端阶段·V3.0 演示用 mock 数据格式
