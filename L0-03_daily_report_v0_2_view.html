INFINITY OS · 门店日报汇总 SKILL · L0-03 v0.2

Skill 范例 · 汇总节点

把一店一天的零散上报,
汇成一份可信日报。

这是依《Agent 设计标准 v1.0》落地的第二个范例:门店日报汇总(L0-03)。它只汇总、不下单 —— 把库存/竞品/动销/客流/活动/已执行揉成一份结构化日报,喂城市经理看板与统治力差距。

依据 Agent 设计标准 v1.0 Backlog L0-03 Workflow · 无 HITL 上游 L0-01 / L0-05 + 文字

v0.2 变更 · 只补这 5 点,旧的对的没动

  1. 1聚合主键 = store + date;一店多美顾 merge 成一份,一美顾多店拆成多份。
  2. 2完全无上报 ≠ 空报:三路全空判为「疑似漏巡」P1 预警,不静默放过。
  3. 3冲突 ≠ 去重:快照与现场矛盾时标 discrepancy 复核,不让陈旧快照盖掉现场。
  4. 4P0 / P1 / P2 预警等级写死成表,不靠临场判断。
  5. 5定义当日 cutoff、迟到并入标记、同日重跑覆盖(幂等)。

§1

角色与目标

唯一职责:把一个美顾在一家店当天的所有零散上报,合并成一份城市经理一眼能扫、数据中台能直接吃的标准化日报。

01
汇总器,不是第二个解析器库存/竞品已有 L0-01 / L0-05 的结构化 JSON。你对账、去重、补缺,不重新从文字解析一遍。
02
只汇总,绝不重复触发实时补货/断货预警在 L0-01 已经发过。你只产汇总视图喂看板,不再发第二次下单。
03
双产出,缺一不可标准 JSON + 人可读摘要两份一起出,过 §10 校验才出件。
04
拿不准就停,不瞎编归一不到的 SKU、矛盾的数据 —— 打标记交人,不替现场做主。

§3

输入:三路汇入

三路都可能缺。文字必到;两个上游 JSON 当天可能没产,对应区块就空,不硬造。

来源形式说明
当日文字语音转写 / 手输美顾一天攒下的口语流水,混了多类信息
L0-01 库存 JSONArtifact库存/断货的真相源,直接读 items
L0-05 竞品 JSONArtifact竞品动作的真相源,直接读 competitor
货架照可选用增强型 LLM 视觉辅助核对陈列/动销
元数据门店 / 美顾 / 日期通道自动带入,构成主键 store+date
同一 (store, date) 可能由多个美顾分别汇入,全部并入同一份;cutoff 之后到达的算迟到,仍并入当日(见 §4)。
§4

处理流程

摄取对账 → 路由 → 链式 → 出件。Step A 是命门,v0.2 的主键/冲突/漏巡都加在这。

三路输入(文字 + L0-01 + L0-05) │ [Step A 摄取·定主键·对账] 命门 │ A0 主键 store+date：一店多美顾→merge 一份 / 一美顾多店→拆多份 v0.2 │ A1 真相源：库存读 L0-01、竞品读 L0-05 │ A2 去重(重复)：文字与上游同 → 不新建条目 │ A3 冲突(矛盾)：快照 vs 现场打架 → 标 discrepancy 复核 v0.2 │ A4 补缺：上游没有的 → 补条,src=free_text,conf 降级 │ A5 漏巡：三路全空 → 疑似漏巡 P1,不静默 v0.2 │ [Step B 路由] 文字剩余 → 陈列 · 动销 · 客流 · 试吃/活动 · 已执行 [Step C 链式] 认货归一(停在归一失败) → 定性化(不伪精确) → 抽数 [Step D 预警] 按 §7.2 等级表写死出 alerts [Step E 出件] JSON + 摘要 → 跑 §10 校验 │ cutoff 后迟到 → 并入当日,标 late_merged,revision++ v0.2 │ 同 (store,date) 重跑 → 覆盖,不新增第二份(幂等) v0.2 │ Output = 标准日报 JSON + 人可读摘要 (Artifact,下游唯一接口)
§5

RFC2119 规则

每条标强度。带 v0.2 的是这版新增。

MUST
库存以 L0-01、竞品以 L0-05 为唯一真相源。文字里重复的必须去重,不得重复建条目。
MUST
断货(out_of_stock)必须进 alerts(P0)并继承 L0-01 标记。
MUST
必须同时产结构化 JSON + 人可读摘要,缺一不出件;sources 可追溯。
MUST
动销/陈列口语 SKU 必须归一;归一不到必须停并打标记,严禁编码。
MUST
主键 = store + date。v0.2一店多美顾 merge 同一份、一美顾多店拆多份,绝不跨店混并。
MUST
三路全空 → 不得当空报静默,必须产「漏巡」预警(P1)。v0.2
MUST
快照与现场矛盾 → 标 discrepancy + 出「数据冲突」alert。v0.2不得用陈旧快照覆盖现场(区别于去重)。
MUST
同 (store,date) 重跑必须覆盖(幂等),迟到并入标 late_merged、revision++。v0.2
MUST NOT
绝不重复触发实时下单/预警。那是 L0-01 的活;本节点扇出只喂看板与分析视图。
SHOULD
客流/动销主观判断应标定性信号,不强行换算成精确数字。
SHOULD
试吃/活动应记份数;摘要按固定段落顺序便于扫读;free_text 补的应降 conf。
MAY
已执行的物料动作(补爆炸卡)可记入 executed,非核心必需项。

§7

扇出 + 预警等级表

7.1 扇出 —— 全是「喂视图」,不是「触发执行」。实时下单/预警在 L0-01 已发生。

条件触发下游性质
永远城市经理看板:写入当日日报视图
competitor 非空统治力差距 Skill分析喂料
loc ∈ (端架,堆头)统治力差距等级 +1分析喂料
inventory.abnormal 非空补货 Skill 汇总视图视图,不重复下单
❌ 反例:看到异常库存又发一次补货触发 —— L0-01 已发过,会重复订货。本节点只提供「今天这家店缺了啥」的快照。

7.2 预警等级判定表 v0.2 —— 写死,不靠临场判断。命中多行取最高(P0 > P1 > P2)。

触发条件等级type说明
status == out_of_stockP0断货营收直接受损,最高优先,继承 L0-01
端架/堆头 scale==largeP1竞品统治力威胁,等级已 +1
discrepancy(快照vs现场)P1数据冲突需人工复核,防陈旧覆盖现场
三路全空P1漏巡巡店覆盖缺口,城市经理需跟进
继承 L0-01 异常 flagP1异常如疑似窜货,透传不自行判定
status == lowP2低库存需补货但不紧急
端架/堆头 scale!=largeP2竞品关注级,不升统治力等级
§8

HITL:本节点【无】

本节点不执行任何钱 / 权限 / 不可逆动作,纯只读汇总,没有可卡的执行点,故无 HITL。

这不是漏写。按《标准 §6》,HITL 卡在「高风险动作执行前」;本节点没有这类动作。真正的 HITL 在下游补货执行节点(大额补货 / 疑似窜货)。本节点只把 L0-01 已打的 hitl_required / flags 如实透传进 alerts,供城市经理知情,不自己停机。

§10

确定性校验(出件前必跑)

能交给脚本的别让 LLM 做。返回非空则不得出件,转 flags。v0.2 新增 4 条断言。

def validate(out):
    errs, seen = [], set()
    # 既有:SKU 存在 / 去重 / 断货必触发 / 双产出 / sources / 不许实时触发
    for it in out["inventory"]["abnormal"]:
        if it["sku"] not in SKU_MASTER: errs += [f"sku_unresolved:{it['sku']}"]
        if it["sku"] in seen:        errs += [f"dup_inventory:{it['sku']}"]
        seen.add(it["sku"])

    # ---- v0.2 新增 ----
    if not out.get("store") or not out.get("date"):
        errs += ["missing_primary_key"]            # ① 主键
    if empty_all(out) and not has_alert(out, "漏巡"):
        errs += ["silent_empty_must_be_missed_patrol"] # ② 漏巡不静默
    for sku in discrepancy_skus(out):
        if not has_conflict_alert(out, sku):
            errs += [f"discrepancy_without_alert:{sku}"] # ③ 冲突成对
    if "late_merged" in out.get("flags", []) and out.get("revision",1) < 2:
        errs += ["late_merge_without_revision_bump"] # ④ 幂等版次
    return errs
