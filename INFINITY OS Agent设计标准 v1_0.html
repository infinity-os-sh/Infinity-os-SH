§1

四条总原则

任何节点都先服从这四条,再谈细节。

01
能简单就别复杂先做最朴素的版本,跑出来再按真实错误加东西。复杂度必须用结果证明自己值得。
02
能写死就别放权步骤可预测的事用 Workflow(轨道),不要交给 Agent 即兴发挥。放权要付出不可控的代价。
03
节点间靠标准化 JSON 硬连接上一节点的 Output 必须是下一节点明确的 Input,不靠"心电感应"。接口稳,系统才稳。
04
高风险动作必有 HITL涉及钱、权限、不可逆操作的,机器必须停机等人拍板。系统要可控,不要黑箱。
§2

第一决策门:Workflow 还是 Agent

拿到任何节点,先回答两个问题。这是整套标准的总开关。

Decision Gate · 决策门

Q1
这事每次都差不多吗?步骤能不能提前画死?
能 / Predictable
→ Workflow
写死的轨道。稳、便宜、可审计。再进 Q2 选模式。
不能 / 要临场判断
→ Agent
放权给 AI 自己决定。灵活但难控,慎用。
Q2
若是 Workflow,这一步在干哪类活?
分类 / 接力
路由 / 链式
先分流到对的窗口;或一道工序接一道。
分发
确定性扇出
一份结果按写死规则同时派给多个下游。

类比:Workflow = 自动扶梯(固定方向,稳);Agent = 出租车司机(自己选路,灵活但你管不住走哪)。库存这类高频规律的事,永远先选扶梯。

§3

模式速查表

五个常用积木 + 两个刻意避开的。"用吗"一栏指库存上报这道标准范例。

模式一句话大白话(配类比)何时用范例用吗
增强型 LLMAugmented LLM配了眼睛+字典+记事本的 AI(会干活的工人)一切的基础积木✓ 用
路由Routing门口分拣员,先分类再处理(医院分诊台)输入混了多种类别✓ 用
链式Chaining一道接一道接力(工厂传送带)步骤有先后依赖✓ 用
确定性扇出Fan-out一份结果按写死规则自动分发(快递格口)下游可预测、要同时触发多个✓ 用
并行Parallelization多份活同时跑 / 同一份跑几遍投票赶速度,或要提高置信度✓ 选用
编排者OrchestratorAI 当包工头,临场决定拆几步、派给谁事先无法预测会冒出哪些子任务✗ 避
线上评估者Evaluator (live)旁边再加 AI 打分、让重做建设期离线用,高频线上别挂✗ 避
避开 Orchestrator / 线上 Evaluator 的原因:库存上报下游永远是补货/断货/竞品那几样,完全可预测 —— 给简单事请包工头、给每条上报配打分员,只会更慢、更贵、更不可控。
§4

设计四步法

从一句人话的 SOP,到一条能上线的生产线。

格式标准化 Standardization停止写散文。把情境转成变量(参数化),用结构化区块切分,并用 RFC2119 给每条规则定强度(见 §5)。
任务拆解 Decomposition把大任务拆成独立节点,每个节点有自己的 Input/Output。节点之间靠标准化 JSON(Artifact)硬连接。哪里坏改哪里。
双向开发 Bi-directional Dev用真实数据挖"默会知识"。别闭门写完美 SOP —— 写粗糙版、跑一次、撞墙、补一条规则,小步快跑迭代。
整合与执行 Integration经 MCP 接真实世界工具(订货系统、数据中台),并在高风险处设 HITL 卡点(见 §6)。
§5

RFC2119 强度规范

每条规则都必须标强度,让节点清楚:哪些不能讨价还价,哪些可以自行判断。

MUST
硬性规定,绝对不能跳过。例:断货必须立刻标记并触发预警;口语 SKU 必须先归一到主数据,归一不到就停下打标记,不许瞎编。
SHOULD
强烈建议,不做需记录原因。例:数量是估算的应标注置信度;竞品堆头应附照片佐证。
MAY
可选项,由节点视上下文决定。例:美顾已执行的动作可记录 —— 对核心决策非必需。
§6

HITL 卡点规范

HITL = Human-In-The-Loop(人在回路)。日常小事机器自己跑;碰到高风险,先停机等人点 OK 才继续。

触发条件动作放在哪
金额超阈值如下单 > 设定额暂停,等人审批执行(下单)前
权限 / 不可逆变更暂停,等人审批执行前
疑似窜货 / 异常暂停,转人工核查执行前
原则:把人放在"高风险动作执行前",不要放在"每条上报前"—— 否则人被淹没,HITL 形同虚设。类比 ATM:取小钱直接给,取大额才二次验证。
§7

标准范例:库存上报 Agent

这是全套标准最干净的一道例题。一条真实上报跑完全程。

美顾小王巡完大润发徐汇店,发来一张货架照 + 语音转文字:「① 六月鲜特级 380ml 货架剩两排大概 8 瓶,② 加点鲜断货了,③ 海天在端架做了金标生抽的大堆头,④ 我把六月鲜的爆炸卡补了。」

第一步 · 拆原子(对应 §4 标准化)

①六月鲜特级380ml 剩约8瓶→库存量(估计,标置信度)
②加点鲜断货→断货(MUST 立即预警)
③海天端架金标堆头→竞品动作(SHOULD 附照)
④已补爆炸卡→已执行动作(MAY 记录)

第二步 · 跑流程(对应 §2 决策门 → Workflow,§3 路由+链式+扇出)

小王原始上报(①~④ + 照片) │ [节点1 解析] 路由分四类 → 链式:认货→查编码→估量→比安全库存 │ Output = 标准 JSON (Artifact,硬连接) │ ├─▶ [节点2 补货] 读 ① status:low → 拟补货单 ├─▶ [节点3 断货预警] 读 ② out_of_stock → 推紧急预警 └─▶ [节点4 竞品战报] 读 ③ competitor → 生成战报、喂统治力差距 │ ↑ 确定性扇出:字段决定派给谁,不靠 AI 临场判断 │ [HITL 卡点] 节点2 补货单 > 阈值 或 疑似窜货 → 停机等人点 OK │ [最终执行] 人批准后才真正下单

第三步 · 标准产出物(节点1 的 Output,下游唯一认的接口)

{
  "store": "大润发徐汇店", "advisor_id": "MG-0420",
  "items": [
    { "sku":"6MX-TJ-380", "qty":8, "conf":"low", "status":"low" },
    { "sku":"6MX-JDX", "qty":0, "status":"out_of_stock" }
  ],
  "competitor": [{ "brand":"海天", "sku":"金标生抽", "loc":"端架", "act":"堆头", "scale":"large" }],
  "executed": ["补爆炸卡"],
  "fanout": ["补货:6MX-TJ-380", "断货预警:6MX-JDX", "竞品战报:海天金标堆头"]
}
默会知识陷阱(§4 双向开发):第一版 SOP 一定漏写"爆炸卡是物料不是产品""端架是高价值位置""两排按货架规格换算"—— 这些靠跑真实数据撞出来,再一条条补进 §5 的 MUST/SHOULD。
