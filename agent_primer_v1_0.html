§0

先理清一件事:「agent」有两层意思

很多混乱都来自这个词撞名。分清楚,后面全通。

广义「agent」
统称 / 一个节点

系统里每个干活的单元都叫 agent(你那 A01–A23)。这层意思上,所有节点都是"agent",没问题。

狭义 Agent
标准定义 / 自主型

特指 LLM 自己决定步骤和工具的那种。这层意思上,绝大多数节点不是 Agent,而是 Workflow。

所以「把 Workflow 都转成 Agent」是错的:目标是把绝大多数做成稳的 Workflow,只有极少数真该放权的才是 Agent。
§1

核心一句话

The one rule

能写死就别放权,能简单就别复杂。

Workflow = 自动扶梯(固定路线,稳);Agent = 出租车司机(自己选路,灵活但你管不住)。高频规律的事,永远先选扶梯。

§2

设计一个节点的全流程(就五步)

从头到尾,顺序固定。任何节点都走这五步。

判 Workflow 还是 Agent Decision Gate能写死→Workflow;不能预测、要临场判断→Agent。这是总开关。
走模式 Pattern若是 Workflow,选它用哪种(常几种组合):分诊=路由、接力=链式、分发=扇出、并跑=并行。
标 RFC2119 强度 MUST / SHOULD / MAY每条规则标清硬度:绝不能跳 / 建议 / 可选。让 AI 知道哪条不能讨价还价。
定 JSON 契约 Artifact约定一张"字段贴好标签的表",当作给下游的输出。节点之间靠它硬连接。
设 HITL 卡点 Human-in-the-Loop高风险动作(钱/权限/不可逆)机器停机等人点 OK,卡在执行前。
§3

名词大白话表

两种身份(先判这个)
Workflow工作流主体
路线提前画死,AI 沿轨道走。稳、便宜、可审计。类比:自动扶梯。
Agent自主体少数
AI 自己决定下一步、用什么工具。灵活但难控,只在步骤无法预测时才用。类比:出租车司机。
基础积木
增强型 LLMAugmented LLM
配了眼睛(读图)、字典(查资料)、记事本(记数据)的 AI。一切的最小单位。类比:会干活的工人。
四种模式(Workflow 选一种或几种)
路由Routing
门口分拣员,先分类再分流。类比:医院分诊台。
链式Chaining
一道工序接一道,前一步的产出交下一步。类比:工厂传送带。
确定性扇出Fan-out
一份结果按写死规则自动分发给多个下游,不靠 AI 临场判断。类比:快递分拣格口。
并行Parallelization
多份活同时跑;或同一份跑几遍取多数,换速度或换置信度。
两个刻意避开的
编排者Orchestrator避
AI 当包工头,临场决定拆几步、派给谁。只在子任务无法预测时才用;可预测的别用,否则又添黑箱。
线上评估者Evaluator避
旁边再加 AI 打分、让重做。放到离线建设期用;高频线上别挂,又慢又贵。
连接与工具
JSON结构化文本
带标签的文本,机器能精准读。类比:一张字段都贴好标签的表。
Artifact产出物
上一个节点吐的那份 JSON,直接当下一个节点的输入。节点之间的"硬连接"。
MCPModel Context Protocol
AI 界的 USB-C,一套标准接口接外部工具(订货系统、数据中台、货架照)。
规则强度 & 把关
RFC2119强度规范
公开技术规范,定义 MUST / SHOULD / MAY 的精确含义,消除"应该"的模糊。详见 §4。
HITL人在回路
高风险动作机器停机等人点 OK。类比:ATM 取大额要二次验证。
方法(怎么真的把它做好)
节点node
流水线上的一个工位。每个 agent / 处理步骤都是一个节点。
四步法4 Steps
格式标准化 → 任务拆解 → 双向开发 → 整合。把一句人话的 SOP 变成能上线的流水线。
默会知识Tacit Knowledge
你脑子里没写下来的"常识",AI 不会通灵。第一版一定漏,要靠跑真实数据撞出来。
双向开发小步快跑
别闭门写完美 SOP。写粗糙版 → 跑 → 撞墙 → 补一条 → 再跑。速度靠"迭代得多快"。
评测Eval
怎么知道它准不准。攒真实样本对照打分,看哪类最常错。从"会搭"到"能上线"的关键一步。
§4

RFC2119 三档(规则的硬度)

它是一份公开技术规范(IETF 1997 年发布,编号 RFC 2119)。原文很短:rfc-editor.org/rfc/rfc2119。你在上传的两份文件和《设计标准》§5 里都见过它。

MUST
硬规定,绝不能跳过。例:断货必须立刻预警;口语 SKU 必须归一,归一不到就停,不许编。
SHOULD
强烈建议,不做要说明原因。例:数量是估算的应标置信度;竞品堆头应附照片。
MAY
可选,做不做都行。例:美顾已执行的动作可记录。

§6

用一个例子串起来:库存上报

五步走完,长这样:

① 高频规律 → Workflow。② 一句话混了几类 → 路由分开,库存这条 链式处理(认货→查编码→估量→比库存)。③ 断货标 MUST、数量估算标 SHOULD。④ 吐一份 JSON 给下游:

{ "sku":"6MX-TJ-380", "qty":8, "status":"low" }

⑤ 这份 JSON 按写死规则 扇出:断货叫断货组、low 叫补货组。大额补货前设 HITL 等人点 OK。

这道题只用到 增强型 LLM + 路由 + 链式 + 扇出 + HITL,刻意没用 Orchestrator 和线上 Evaluator —— 所以最干净,适合当第一道练手题。
