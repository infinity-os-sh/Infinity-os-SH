Step 1·2

数据结构 + 晶体管 DB

L1
两层:事实层(原始度量,T+1 从汉询/POS/SFA 灌)+ 晶体管层(派生的二元状态,由门引擎算出)。晶体管 = 「对象 × 某店 × 某时」的状态,对象可以是 SKU / 区隔 / 品类——用 entity_type 区分,这一列就是防混层的硬约束。
事实层 · inf_fact_store_sku(原始度量,含心跳 r)

Step 1·2

两本账 = 同一张表两种聚合

不是两套数据
两本账不复制数据,只是对 inf_fact / inf_transistor 的两个聚合视图,维度不同、数据不失真(同你 v8.2 一本账双入口的思路)。
视图 vw_scorecard_xinhe · 账 X(欣和 SKU)
按 店 × 欣和SKU × 六卡角色 聚合。算:r 实际 vs node_target → 缺口(自上而下,不锚实际)。数据齐(汉询/SKU状态/目标),V0 可直接跑真数据。
视图 vw_scorecard_retailer · 账 A(零售商品类)
按 店 × 品类/区隔 聚合全品牌(含竞品)。算:Fair Share → Point Gap → 机会(剔衰退)。竞品/整店数据缺 → 低置信,靠一线 INF-XD 回填、第三方大盘校。
对应门(keystone)就是把这两个视图按 店×区隔 join:零售商的某区隔有缺口(账A) ∩ 欣和有能补的 SKU(账X)→ 交集处出唯一动作。两本账各自的"角色"列不互相写,结构上杜绝混层。
Step 3

逻辑门引擎(统一契约)

L2 · 门只插不补
7 道门全实现同一个接口,插进一个引擎。门分三种:TABLE(查表,如角色→战术)、FORMULA(算式,如 Fair Share)、MODEL(模型,留给因果门)。规则即数据:门的内容是配置,不是硬编码——改业态/改规则只改配置。
统一契约 · Gate
interface Gate {
  String id();
  GateKind kind();              // TABLE | FORMULA | MODEL
  GateResult evaluate(GateContext ctx);
}
record GateContext(TransistorState st, Metrics m,
                   ChannelParams l4, Map<String,Object> ext) {}
record GateResult(String gateId, Verdict verdict, // PASS|FAIL|FLAG
                  Double gapValue, List<String> evidence,
                  Confidence confidence,           // REAL|APPROX|LOW
                  ActionHint action) {}
门 #1 角色→4战术 · TABLE 型(规则即数据,搬自 Spec §5)
gate: role_to_tactics      kind: TABLE
lookup:
  优先性: { range: Broad,    price: StronglyCompetitive,
            shelf: Quality,  promo: Build }
  基本性: { range: Basic,    price: Competitive,
            shelf: Average,  promo: Maintain }
  目的性: { range: Complete, price: Aggressive, ... }
  # 季节性 / 便利性 同表
门 #2 Fair Share · FORMULA 型
# 理想:第三方品类大盘
fair_share = retailer_size_in_region / market_size_in_region
# V0 近似(缺大盘时)→ confidence=LOW
fair_share ≈ 欣和该品类在该店份额基准
point_gap   = actual_share − fair_share
opportunity = Σ[区隔: 增长=1 AND point_gap<0] |point_gap| × seg_size
门 #7 因果门 V0 只留接口(kind=MODEL 的空实现),前 6 道门跑出的每个执行实例持续喂它样本,养而非速成。
