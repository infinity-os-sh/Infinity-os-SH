1. 关系定位

┌─────────────────────────────────────────────────┐ │ INFINITY OS · 全集团经营 OS │ │ │ │ ┌─────────────────────────────────────────┐ │ │ │ L1 战略层 │ │ │ │ ┌────────────────────────────────────┐ │ │ │ │ │ BAS (六月鲜) · 战略事实来源 ⭐ │ │ │ │ │ │ 未来: BAS-禾然 / BAS-味达美... │ │ │ │ │ └────────────────────────────────────┘ │ │ │ └─────────────────────────────────────────┘ │ │ ↓ 读取战略信息 │ │ ┌─────────────────────────────────────────┐ │ │ │ L2-L4 经营层 · 24 Agent │ │ │ │ Agent 从 BAS 读取, 不重复定义战略 │ │ │ └─────────────────────────────────────────┘ │ │ ↓ │ │ ┌─────────────────────────────────────────┐ │ │ │ L5 市场单元 · 城市/区域级执行 │ │ │ └─────────────────────────────────────────┘ │ └─────────────────────────────────────────────────┘

核心原则: BAS 是 INFINITY OS 各 Agent 的战略上游, 不是平级模块. 任何涉及"品牌战略/SKU 阶段/副品牌定位/心智状态"的 Agent 决策, 都从 BAS 读取, 不在 Agent 内部硬编码.

2. 8 个连接点 · 详细 Schema

连接点 1 · BAS L4 SKU → 智能补货 Agent (spec_03)

BAS 提供: SKU 阶段 + 主推权重 + 角色

Agent 用于: 计算补货量 + 补货优先级

调用方式: Agent 启动时读 BAS, 每日刷新

GET /bas/v1/skus/{sku_id} Response: { "id": "SKU-05", "name": "遵循自然零添加+轻盐 500ml", "stage": "S2", // S0-S6 "role": "main_push", // main_push / cash_cow / defend / new "income_share": 0.15, // 美顾激励权重 "primary_l2": "L2-2", "trend": "strengthening", // strengthening/stable/declining/starting "consumer_value": { "spec": {...}, "product": {...}, "brand": {...} } }

下游逻辑示例:

  • S0 孵化 → 推广性补货 (+30% 安全库存)
  • S3 成熟 → 标准 a 值补货
  • S5 衰退 → 减量补货 (-20%)
  • main_push 角色 → 优先满足缺货

连接点 2 · BAS 心智趋势 → 蝴蝶效应 Agent (spec_02)

BAS 提供: 心智沉淀趋势信号 (↗ / → / ↘ / ⟳)

Agent 用于: 跨节点失真检测加权 · 战略性失真识别

GET /bas/v1/mind-trends Response: { "L2-1": { "trend": "mixed", "ast_changes": [...] }, "L2-2": { "trend": "strengthening", "key_risk": "digestion_rate" }, "AST-01": { "trend": "declining", "rate": -0.09 }, "AST-02": { "trend": "strengthening" }, ... }

下游逻辑示例:

  • 心智 ↘ 稀释 + 销量 → 稳定 = 压货/促销假繁荣 (B 类失真)
  • 心智 ↗ 强化 + 消化率 < 50% = 压货透支心智 (新失真类型)
  • 跨副品牌迁移信号检测 (经典 → 遵循自然)

连接点 3 · BAS 副品牌阶段 → A 值校准 Agent (spec_01)

BAS 提供: 每个副品牌的阶段 (S0-S6) + 阶段转换规则

Agent 用于: 不同阶段使用不同 a 值校准规则

A 值校准规则 (按阶段): S0 (孵化): a = 1.0 × seasonal_factor (基础假设) S1 (引入): a = recent_30d_avg × 1.2 (爆发期向上修正) S2 (成长): a = recent_30d_avg × 1.0 (标准) S3 (成熟): a = trailing_90d_avg (稳定) S4 (衰退): a = recent_30d_avg × 0.9 (向下修正) S5 (衰退): a = recent_30d_avg × 0.8 S6 (退场): a = recent_30d_avg × 0.5 (清库存)

连接点 4 · BAS income_share → Uber 激励 Agent

BAS 提供: SKU 美顾激励权重 + 主推标志

Agent 用于: 美顾派单优先级 + 任务奖励计算

SKU-05 income_share = 0.15 (vs 平均 0.10) → 美顾 APP 显示: "重点主推" 标志 → 货架照片识别完成后, 奖励 = 标准 × 1.5 → 月度结算优先级提升

连接点 5 · BAS 主推 SKU 矩阵 → 派单互锁 25 步

BAS 提供: 各副品牌的主推 SKU 清单 + 优先级

Agent 用于: 美顾派单时显示的 SKU 排序 + 任务模板

派单互锁第 6-9 步 (SFA 显示阶段) 从 BAS 读取要主推哪些 SKU.

连接点 6 · BAS 副总监仪表盘 → 主控台个人工作台

BAS 提供: 5 副总监各自的仪表盘视图

主控台用于: 副总监登录后, "个人工作台"嵌入 BAS 仪表盘 (作为 iframe 或原生组件)

主控台路由: /dashboard/sub-manager/{manager_id} → 内嵌 BAS 仪表盘组件 → 共享认证 (SSO) → 实时显示该副总监的 KPI/SKU/协议

连接点 7 · BAS 协议层 → 资源分配 (Cash 脉络)

BAS 提供: 6 份协议 + 执行进度

INFINITY OS 用于: 跨副品牌资源转移的实际执行

P-01 经典 → 遵循自然 ¥3,000万营销资源 INFINITY OS 实际执行: - 财务系统: 预算从 L2-1 转 L2-2 - 营销系统: 媒介预算切换 - 派单系统: 美顾任务结构调整 - 一本账: 资源消耗记录

连接点 8 · BAS 起量奇迹案例库 → INFINITY OS L6 记忆

BAS 提供: AST-07 + CT-07 案例库 (结构化因果链路)

INFINITY OS 用于: 跨品牌学习层共享 · 新品孵化决策辅助

这是集团级 IP, 不只服务六月鲜, 也服务禾然/味达美等品牌的新品孵化.

  • SKU 营收/利润: T+1 (每日)
  • 消化率/库存: T+1

4. 4 阶段嵌入路线图

🔵 阶段 1 · 准备期 (本月 - 2 个月)
条件: 无 (现在就能做)

动作:

  • BAS v0.9 部署到 GitHub Pages (独立运行)
  • 5 副总监试用 · 收集反馈
  • SAP Step 2 解锁 (业务团队推) — 这是阶段 2 的前置
  • Tech Lead 招聘到位 (用现有 toolkit)

不做: 任何整合工作 · BAS 跟 INFINITY OS 完全独立

🟢 阶段 2 · 数据对齐 (3-6 个月)
条件: SAP Step 2 完成 · BAS v1.0 (业务反馈版本) 完成

动作:

  • BAS 数据后端搭建 (MySQL + Spring Boot, 跟主控台共享)
  • SKU_VALUES / L2_VALUES / SCN_VALUES 真实数据替换
  • BAS API 开发 (10 个核心 endpoint)
  • 消化率/复购率数据接入

不做: Agent 还没读 BAS · UI 仍分离

🟡 阶段 3 · 接口对接 (6-12 个月) ⭐ 真正嵌入开始
条件: BAS API 稳定 · 至少 1 个 Agent 真跑 (不是 Layer 1.3)

动作 (按优先级):

  1. 连接点 1 (智能补货 ← SKU 阶段) — 最直接
  2. 连接点 4 (Uber 激励 ← income_share) — 已有原型
  3. 连接点 6 (副总监仪表盘 ← 主控台) — UI 整合
  4. 连接点 2 (蝴蝶效应 ← 心智趋势)
  5. 连接点 5 (派单互锁 ← 主推 SKU)
  6. 连接点 3 (A 值校准 ← 阶段)
  7. 连接点 7 (资源分配 ← 协议层)
  8. 连接点 8 (L6 记忆 ← 案例库)

效果: Agent 决策开始基于 BAS 战略, 不再硬编码

🟣 阶段 4 · 全集团扩展 (12-24 个月)
条件: 六月鲜 BAS 跑通 · 业务验证有效

动作:

  • BAS-禾然 复制建模 (用六月鲜 BAS 作为模板)
  • BAS-味达美 / BAS-小康 / BAS-葱伴侣
  • 集团 BAS 统一视图 (5 品牌跨视角)
  • 跨品牌接力机制实际运行 (P-05 Origin Lab → 禾然顶级)
  • L5 市场单元延伸 (城市/区域级 BAS)

5. 技术栈对齐

跟 INFINITY OS 主控台 v8.4 一致

  • 后端: Java Spring Boot + MySQL + Redis
  • 数据源: Hologres (主查询) + MaxCompute (历史) + 汉询 (rixiao)
  • 前端: HTML + 渐进升级 (后续考虑 React)
  • 认证: SSO 跟主控台共享
  • 权限: L1/L2/L3/L4 + 副总监 5 级
  • API 风格: RESTful + JSON
  • 部署: 阿里云 + GitHub Pages (预览)

6. 关键风险

⚠ 风险 1 · 时机过早嵌入
BAS 内容还没业务验证就嵌入 → 错的假设直接驱动 24 Agent → 系统性错误.
缓解: 严格按 4 阶段, 阶段 2 前不做整合.
⚠ 风险 2 · 数据后端碎片化
BAS 自己建数据库, 跟主控台不一致 → 战略数据 vs 运营数据分裂.
缓解: 阶段 2 共享后端 (MySQL + Hologres).
⚠ 风险 3 · 战略层频繁变动
BAS 内容变动太快, Agent 跟不上 → 决策混乱.
缓解: BAS 版本化, 每个版本对应一组 Agent 配置.
⚠ 风险 4 · 跨品牌定位冲突
BAS-六月鲜 + BAS-禾然 + ... 之间定位重叠/冲突.
缓解: 集团 BAS 层 (阶段 4) 协调跨品牌策略.
